# SGF 專案進度控制台

SGF 專案進度控制台是一個以 Google Sheet 為資料來源的專案管理儀表板。使用者可在瀏覽器中查看任務進度、依條件篩選任務、編輯任務內容，並透過 Google Apps Script Web App 將變更同步回 Google Sheet。

## 功能概覽

- 顯示任務總數、已結案數、未結案數與整體平均完成度。
- 以分類手風琴呈現任務，可全部展開、全部折疊或聚焦單一分類。
- 搜尋任務名稱與目標／備註內容。
- 依主要負責人、支援角色、結案狀態、上週／當週／下週及特殊標記篩選。
- 支援優先任務（`>`）、等待 Andy 確認（`!!`）及進度以 `-` 開頭的任務篩選。
- 由側邊編輯抽屜修改任務名稱、ID、負責人、協作人員、完成狀態、連結、備註與週進度。
- 新建任務，並將任務資料寫入 Google Sheet。
- 顯示週進度歷程與任務時間軸。
- 顯示「時程」工作表的每日里程碑，可依月份或週別瀏覽及編輯。
- 手動觸發 Google Sheet 的 `manualSortTasks` 排序後重新載入。
- API 連線網址與金鑰儲存在瀏覽器 `localStorage`，方便同一台裝置下次使用。
- 頁尾提供可自訂的隨機短句，僅儲存在瀏覽器本機。

## Google Sheet 資料格式

後端預期同一份試算表至少包含以下工作表：

### `Task`

前兩列是欄位標題。週進度欄位使用 `W` 加數字命名，例如 `W27`、`W28`；第二列可放該週的日期區間，例如 `06/29 - 07/05`。一般欄位名稱包括：

| 欄位 | 用途 |
| --- | --- |
| `完成` | 核取方塊，代表是否結案 |
| `ID` | 任務識別碼；若沒有明確 `ID` 欄位，程式會使用任務名稱前一欄 |
| `專案項目` | 任務名稱，可使用超連結 |
| `主` | 主要負責人 |
| `副` | 支援／協作人員欄位，可有多欄 |
| `目標 (需要文件)` | 任務目標、備註或文件需求 |
| `完成度` | 0～1 或百分比數值 |
| `Wxx` | 各週進度內容 |

以 `//` 開頭的專案項目會被視為分類標題；分類標題下方的任務會歸入該分類。

### `時程`

後端讀取 C 欄日期、D 欄星期、F 欄里程碑目標。前端可瀏覽指定月份或週別；已有資料的日期可直接編輯並同步回 F 欄。

## Google Apps Script 設定

1. 開啟作為資料來源的 Google Sheet。
2. 進入「擴充功能 → Apps Script」。
3. 將 `google-apps-script.js` 全部貼入 Apps Script 專案。
4. 確認試算表包含 `Task` 與 `時程` 工作表，且欄位名稱符合上方格式。
5. 在程式中設定 `API_KEY`，不要使用公開或容易猜測的金鑰。
6. 部署為「網頁應用程式」：以本人執行，存取權限依團隊需求設定。
7. 複製部署後的 `/exec` URL。
8. 在控制台右上角開啟設定，填入 Web App URL 與相同的 API Key。

## 本機執行與部署

這是靜態網站，不需要安裝相依套件。可直接使用任一靜態檔案伺服器或部署到 GitHub Pages、Netlify、Cloudflare Pages 等服務。

若使用 Python，可在專案根目錄執行：

```bash
python -m http.server 8000
```

然後開啟 <http://localhost:8000>。直接以 `file://` 開啟也可能顯示頁面，但部分瀏覽器的跨來源限制可能影響 API 連線，建議使用 HTTP 靜態伺服器。

若環境沒有 Python，可使用專案內的 PowerShell 靜態伺服器：

```powershell
powershell -ExecutionPolicy Bypass -File .\serve.ps1
```

同樣開啟 <http://localhost:8000>。`serve.ps1` 只提供本機測試用的靜態檔案，不包含正式部署功能。

## API 說明

所有請求都需要在 query string 帶入 `key`：

```text
GET  {WEB_APP_URL}?key={API_KEY}
POST {WEB_APP_URL}?key={API_KEY}
```

### GET

`doGet()` 讀取 `Task` 與 `時程`，回傳：

```json
{
  "tasks": [],
  "milestones": [],
  "weeksList": []
}
```

### POST 動作

| `action` | 用途 |
| --- | --- |
| `createTask` | 新增任務列，並沿用上一列的公式 |
| `updateTask` | 更新既有任務資料與週進度 |
| `updateMilestone` | 更新 `時程` 工作表的里程碑目標 |
| `sortTasks` | 執行 `manualSortTasks()` 並重新排序任務 |

前端以 `Content-Type: text/plain` 傳送 JSON，這是為了降低 Google Apps Script Web App 的跨來源請求問題。後端會回傳 JSON，成功時包含 `success: true`。

## 使用流程

1. 開啟頁面並確認 API 設定。
2. 儀表板載入 Google Sheet 的最新資料。
3. 使用上方統計卡、搜尋框與篩選條件定位任務。
4. 點擊任務右側編輯按鈕修改內容。
5. 儲存後，前端先更新畫面，再將資料寫回 Google Sheet。
6. 若需要依規則重排資料，點擊重新整理／排序按鈕。

## UI／UX 設計

### 設計定位

介面以「專案進度控制台」與報紙編排風格為核心，將高密度的任務資料整理成可快速掃讀的工作畫面。整體採用編輯感較強的標題字體、細線分隔、卡片區塊與紅色強調色，營造正式但具辨識度的專案管理工具風格。

### 視覺系統

- **色彩**：以暖白／淺色背景與深色文字為基礎，使用紅色作為主要操作、警示、今日日期與優先狀態的強調色。
- **元件**：統計卡片、篩選晶片、任務手風琴、側邊編輯抽屜與 Modal 皆使用一致的邊框、圓角、陰影與半透明玻璃卡片風格。
- **字體**：主要內容使用 Inter／Outfit 類無襯線字體，標題與日期資訊搭配 Playfair Display，區分資訊層級並強化報紙風格。
- **圖示**：使用 Font Awesome 圖示輔助辨識搜尋、設定、重新整理、日曆、編輯、完成等操作。
- **狀態色**：已完成、進行中、優先任務、等待確認與錯誤訊息都有獨立的視覺提示，降低使用者閱讀表格文字的負擔。

### 頁面資訊架構

```text
頁首
├─ 品牌／系統名稱
├─ 年度、週次、今日日期
└─ 時程、說明、重新整理／排序、API 設定

主要內容
├─ 統計摘要：總任務、已結案、未結案、整體進度
├─ 篩選工具列：搜尋、負責人、角色、狀態、時間、特殊標記
└─ 任務列表：分類手風琴 → 任務列 → 編輯操作

輔助操作
├─ 專案時程／里程碑 Modal
├─ 任務編輯側邊抽屜
├─ API 設定 Modal
├─ 使用說明 Modal
└─ Toast 操作結果提示
```

### 核心互動設計

1. **先摘要、後細節**：使用者進入頁面先看到四項統計，再透過篩選器縮小任務範圍，最後進入分類與任務明細。
2. **手風琴降低資訊負擔**：任務按分類收合，支援全部展開、全部折疊與單一分類聚焦，適合處理大量工作項目。
3. **篩選條件即時回饋**：搜尋與篩選不需重新載入頁面；結果數量會同步更新，特殊篩選啟用時會調整不相容的狀態／時間條件。
4. **編輯不離開上下文**：任務透過側邊抽屜編輯，保留原本列表的上下文；儲存後先更新本地畫面，再同步至 Google Sheet。
5. **破壞性或不確定操作提供保護**：編輯內容有變更時，關閉抽屜會先判斷是否存在未儲存變更；儲存、失敗與連線錯誤以 Toast 回饋。
6. **日期與週次雙視角**：時程 Modal 支援月份檢視與週別檢視；今日日期以強調色標示，便於每日工作追蹤。

### 任務列表設計

任務列以固定欄位排列，主要包含：完成狀態、任務名稱、負責人／協作者、目前週進度、完成百分比與編輯按鈕。任務名稱可顯示外部連結；`>` 與 `!!` 等特殊標記會轉換為醒目的標籤，避免重要任務被一般文字淹沒。

分類列會顯示該分類的任務數與平均進度條，讓使用者能在展開前先掌握分類健康度。

### 響應式與操作體驗

- 使用 CSS media query 調整桌面與窄螢幕下的欄位、卡片與抽屜版面。
- Modal 與抽屜開啟時鎖定背景捲動，避免使用者誤操作頁面底層內容。
- 編輯抽屜在小螢幕上改以較適合垂直閱讀的單欄呈現。
- 按鈕提供文字、圖示與 `title` 提示；主要操作集中於頁首與任務列表標題列。
- 載入中、無資料、無符合結果、成功與錯誤狀態皆有明確提示。
- 里程碑編輯支援 `Ctrl + Enter` 儲存與 `Esc` 關閉，提升鍵盤操作效率。

### UX 注意事項與後續優化方向

- 目前篩選條件主要保存在記憶體中，重新整理頁面後會回到預設值；若需要持續工作，可將篩選狀態同步到 URL 或 `localStorage`。
- 大量任務目前以 HTML 一次渲染，若資料量持續成長，可考慮虛擬列表或分頁。
- 可補充更完整的鍵盤焦點管理、ARIA label、錯誤重試按鈕與 API 連線狀態指示，以提升無障礙與故障排除體驗。
- 可將目前散落於 `index.html` 的 inline style 抽離至 `style.css`，讓視覺 token、元件狀態與響應式規則更容易維護。

## 多視圖主題架構

目前已預留主題獨立視圖：

```text
主題切換器／主題側欄
├─ theme-view-sgf       # 現有 SGF 控制台，維持原本 API 與互動
├─ theme-view-theme2    # 部門任務雷達，可獨立改成看板或表格
└─ theme-view-theme3    # 交付里程碑，可獨立改成時間軸或甘特圖
```

主題切換只控制各視圖的顯示狀態，不會以新主題的內容覆蓋 SGF 主題 DOM。因此後續可以分別替換主題二、三的版面、統計、篩選器與資料服務，而不需要修改主題一的既有事件與 Google Sheet 流程。

新增正式主題時，建議遵循以下邊界：

- 使用自己的視圖容器與 CSS 命名空間。
- 使用自己的 `renderThemeX()` 與篩選狀態。
- 使用自己的資料轉換器，最後再接到共用的標準資料格式。
- 不直接讀寫主題一的 `appState`、DOM 節點或事件綁定。
- 共用的部分只保留主題選單、連線提示、Toast 與整體設計 token。

## 安全與維護注意事項

- API Key 會被前端送出，且會儲存在瀏覽器 `localStorage`；它不是嚴格意義上的秘密。若資料敏感，應在 Google Apps Script 或其他後端加入更完整的身分驗證與權限控管。
- `google-apps-script.js` 與前端的 API Key 必須一致；變更後端金鑰後，使用者也要重新在右上角設定。
- 修改 Google Sheet 欄位名稱時，必須同步調整 Apps Script 中的欄位查找邏輯。
- `index.html` 目前載入的是 `app_v10.js`，修改 `app.js` 不會影響線上頁面。
- 目前未提供自動化測試或建置流程；修改後應至少手動驗證讀取、篩選、新增、編輯、里程碑與排序功能。

## 已知限制

- Google Apps Script Web App 的權限、部署版本與跨來源行為會直接影響前端連線。
- 使用者可見的篩選與儀表板統計是在瀏覽器端計算，資料量很大時可能影響渲染效能。
- 多人同時編輯同一列時，後儲存者可能覆蓋先儲存者的變更。
